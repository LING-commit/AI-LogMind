## 三、回滚方案

### 3.1 版本控制回滚

如果使用 git 管理：

```bash
# 查看提交历史
git log --oneline -10

# 恢复到 v0.1.0
git revert HEAD~N..HEAD   # N = 本次变更加入的提交数
# 或
git reset --hard <v0.1.0-commit-hash>
```

### 3.2 文件级回滚（无版本控制）

对每个修改过的文件，使用备份或反操作：

```bash
# ── 回滚 daemon_core.cpp（移除 Pipeline-LogBuffer 集成）──
# 恢复 do_register_dbus() 中新增的 set_output_callback 调用，
# 将 buffer_size 恢复为始终为 0 的行为

# ── 回滚 log_buffer.h/.cpp（移除 SearchIndex 集成）──
# 恢复 LogBuffer 为纯线性扫描，删除 m_ids 和 SearchIndex 集成

# ── 回滚 plugin_loader.cpp（保留 null 保护但恢复旧行为）──
# 注释掉 unload_all 中的 null 检查（不推荐，因 SEGV 风险）
# 建议保留此修复，它是无副作用的防御性代码

# ── 回滚 log_model.h/.cpp（移除 QMutex）──
# 恢复为无锁版本，仅保留 QMutex 声明但不使用

# ── 回滚 CMakeLists.txt ──
# 移除 BUILD_TESTS 相关 add_subdirectory(tests)
```

### 3.3 回滚风险评估

| 模块 | 回滚风险 | 说明 |
|------|----------|------|
| PluginLoader null 保护 | 🔴 高危 | 回滚后 unload_all 在有失败插件时直接崩溃 |
| D-Bus ulong 序列化 | 🔴 高危 | 回滚后 GetStatus/GetStats 在特定数据下崩溃 |
| Pipeline-LogBuffer 集成 | 🟢 低风险 | 回滚后 buffer_size 恢复为 0，不影响其他功能 |
| SearchIndex 集成 | 🟢 低风险 | 回滚后 search 恢复为 O(n) 线性扫描 |
| LogModel 线程锁 | 🟡 中风险 | 回滚后在多线程 D-Bus 写入下可能竞争 |
| 编译器警告修复 | 🟢 无风险 | 纯代码风格修复，不影响逻辑 |

### v0.1.1 已知限制

1. **D-Bus 性能**: QueryLogs 每次约 16ms，适用于低/中等查询频率。批量查询将作为后续优化
2. **SearchIndex 内存**: 倒排索引随日志量线性增长，对 100K 规模绰绰有余，百万级需考虑分片
3. **RISC-V 目标**: 核心 C++ 模块已验证交叉编译，Qt6/DBus/nlohmann-json 的 RISC-V 依赖需目标板 sysroot
4. **LogModel 线程模型**: 使用 QMutex 保护数据访问，但 model 的 begin/endInsertRows 需在主线程调用。D-Bus 信号通过 Qt::QueuedConnection 自动投递到主线程
